# 01. Core Questions and Open Issues

이 문서는 분석/설계 진행 중 사용자, 교수자, 또는 도메인 담당자에게 확인해야 할 질문과 현재 결정 상태를 기록한다.

## Decision Status Legend

- `Resolved`: 출처 또는 사용자 확인으로 결정됨.
- `Developer Decision`: 고객/출처가 정하지 않았으므로 개발자가 설계 결정으로 처리.
- `Open / Needs Discussion`: 출처에 없고 해석 차이가 커서 추가 논의 필요.

## 1. 주간 가용 자금 산식

### Q-001. 사용 가능한 금액 산식의 정확한 부호는 무엇인가?

원문은 다음 항목을 사용한다고 설명한다.

1. 주간 예상 투자 수익
2. 주간 예상 운영 비용
3. 해당 주의 총 예상 모기지 상환액
4. 해당 주의 총 예상 교부금

결정된 산식:

```text
available = weekly investment income
          - weekly operating expenses
          + expected mortgage repayments
          - expected grants
```

해석:

- 투자 수익은 가용 자금을 증가시킨다.
- 운영 비용은 가용 자금을 감소시킨다.
- 수혜자로부터 받을 것으로 예상되는 모기지 상환액은 가용 자금을 증가시킨다.
- 재단이 지급해야 하는 예상 교부금/보조금은 가용 자금을 감소시킨다.

상태: `Resolved`

## 2. 100% 모기지 승인 조건

### Q-002. 90% 모기지 조건은 사전 자격 판단에만 쓰이는가?

MSG Foundation은 100% 모기지를 부여하지만, 조건 중 “90% 모기지에 대한 할부금 합산이 총소득의 28%를 초과하는 경우”가 포함되어 있다.

결정:

- 90% 모기지 조건은 파일럿 시스템에서 직접 계산하거나 관리하지 않는다.
- 수혜자 자격 심사와 선정은 시스템 외부에서 사람이 오프라인으로 판단한다.
- 파일럿 시스템은 주간 자금 계산에 필요한 투자 데이터, 운영 비용 데이터, 모기지 데이터만 관리한다.

상태: `Resolved — Out of Pilot`

## 3. 신규 신청 심사 범위

### Q-003. 파일럿 시스템은 신청 승인/거절까지 판단해야 하는가?

요구사항은 “주간 자금 계산에 필요한 데이터 항목만” 포함하라고 한다. MSG Foundation의 모기지 부여 조건에는 결혼 기간, 고용 증거, 지역 주택 중간 가격, 저축액 등이 포함되지만, 이는 파일럿 시스템의 직접 기능이 아니다.

결정:

- 파일럿 시스템은 자격 심사, 신청 승인/거절, 수혜자 선정 기능을 포함하지 않는다.
- 해당 판단은 시스템 밖에서 사람이 수행한다.
- 파일럿 시스템의 책임은 이미 관리 대상이 된 모기지/투자/운영비 데이터를 기반으로 주간 가용 자금을 계산하는 것이다.

상태: `Resolved — Out of Pilot`

## 4. 보고서 출력 형태

### Q-004. 보고서는 화면 표시, 파일 출력, 인쇄 중 무엇을 목표로 하는가?

원문은 “요청 시 출력(print on request)”만 명시하며 구체적인 출력 매체를 지정하지 않는다.

결정:

- 보고서의 화면 표시, 파일 출력, 인쇄 가능 레이아웃 등 구체적 방식은 개발자 재량으로 설계한다.
- 파일럿 문서는 보고서 중심 UI와 인쇄 가능한 화면/텍스트 출력이 가능한 구조를 기본 제안으로 둔다.

상태: `Developer Decision`

## 5. 데이터 갱신 주기와 책임

### Q-005. 분기별 갱신 데이터는 누가 입력하는가?

투자 예상 수익과 운영 비용은 보통 연 4회 갱신된다고 설명하지만, 갱신 담당자가 수혜자 본인인지, 재단 관리자/투자 고문인지 명확히 결정되어 있지 않다.

결정:

- 파일럿 시스템에서는 복잡한 권한 모델을 도입하지 않고 재단 관리자 또는 승인된 운영 담당자가 데이터를 입력/갱신하는 관리자 중심 모델을 사용한다.
- 투자 고문은 데이터를 제공하거나 검토할 수 있으나, 직접 시스템 사용 여부는 구현 시 선택 가능한 역할로 둔다.
- 과거 전체 이력 보존은 요구사항이 아니며, 각 데이터에는 `Date last updated`만 유지한다.

상태: `Developer Decision`

## 6. 금액 정밀도와 반올림

### Q-006. 금액 계산의 반올림 규칙은 무엇인가?

모기지, 세금, 보험, 보조금, 가용 자금 계산은 금액 단위를 다룬다. 센트 단위 반올림 방식이나 주간 1/52 환산 시 발생하는 소수점 처리 규칙은 출처에 명시되어 있지 않다.

개발자 가정안:

- 모든 금액은 decimal/fixed-point 방식으로 계산한다.
- 보고 및 저장 금액은 통화 단위에 맞춰 센트 단위, 즉 소수점 둘째 자리까지 반올림한다.
- 연간 금액을 52로 나누어 주간 금액을 만들 때도 최종 표시/저장값은 센트 단위로 반올림한다.

상태: `Developer Decision — Source Not Specified`

## 7. 날짜와 주간 기준

### Q-007. “매주 초”와 “매주 말”의 기준일은 언제인가?

주간 계산과 투자 시점이 요구사항에 등장하지만, 구체적인 기준 요일은 출처에 명시되어 있지 않다.

결정:

- “매주 초”는 해당 주의 첫 영업일로 정의한다.
- 영업일은 서비스 제공 국가의 공휴일과 재단 휴업일을 제외한 평일(월~금)이다.
- “매주 말”은 해당 주의 마지막 영업일 마감 시점으로 정의한다.

상태: `Resolved by Project Decision`

## 8. 기존 모기지의 예상 상환액

### Q-008. 총 예상 모기지 상환액은 모든 활성 모기지의 주간 P&I와 고객 부담금을 포함하는가?

모기지 데이터에는 `Weekly principal and interest payment`, `Current combined gross weekly income`, 세금, 보험료가 있다. 알고리즘에서는 “총 예상 모기지 상환액”과 “총 예상 교부금”을 별도 단계로 계산한다고만 되어 있어, 상환액의 정확한 범위는 명확하지 않다.

가능한 해석:

1. `expected mortgage repayments` = 수혜자가 실제로 재단에 납부할 것으로 예상되는 금액.
2. `expected mortgage repayments` = 보조금 지급 전 P&I + 에스크로 전체 비용.

권장 해석:

- 현재 결정된 산식이 `+ expected mortgage repayments - expected grants` 구조이므로, 중복 계산을 피하기 위해 `expected mortgage repayments`는 수혜자로부터 실제 받을 것으로 예상되는 상환액으로 해석하는 것이 가장 안전하다.
- 단, 출처가 명확하지 않으므로 최종 구현 전 추가 논의가 필요하다.

상태: `Open / Needs Discussion`

## 9. 시스템 형태

### Q-009. 파일럿 시스템의 기대 실행 형태는 무엇인가?

과제 요구사항에는 구체적인 플랫폼이 없다.

결정:

- CLI, Web, Desktop 등 구현 플랫폼은 개발자 재량으로 결정한다.
- 분석/설계 산출물은 특정 플랫폼에 과도하게 종속되지 않되, 구현 시 단순한 보고서 중심 UI를 우선 제안한다.

상태: `Developer Decision`

## 10. 평가 기준

### Q-010. 교수자/과제 평가에서 필수 산출물 형식이 있는가?

결정:

- 필수 산출물은 시스템 분석 및 설계 문서이다.
- 유스케이스, 도메인 모델/클래스 다이어그램, 시스템 시퀀스 다이어그램, 오퍼레이션 컨트랙트 등이 포함된다.
- 최종 산출물은 결과물뿐 아니라 개발 과정에서 겪은 문제와 해결 과정이 담긴 기술 문서 형태여야 한다.
- 코딩/구현은 필수 산출물이 아니라 추가 점수 획득을 위한 선택 사항이다.

상태: `Resolved — Analysis and Design Scope`

## 11. 부분적 재량: 모기지 보조금 및 에스크로 비용 처리

수혜자가 내야 하는 에스크로 비용(세금, 보험료 등)을 용도별로 엄격히 분리할지, 주 단위로 단순 합산할지는 출처에서 결정하지 않았다.

개발자 가정안:

- 파일럿 시스템에서는 에스크로 비용을 세금/보험료 등으로 보고서에는 표시하되, 계산 단계에서는 주 단위 모기지 관련 총비용에 포함하여 단순 합산한다.
- 향후 확장을 위해 원천 필드는 세금과 보험료로 보존한다.

상태: `Developer Decision — Source Not Specified`
